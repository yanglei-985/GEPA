in one line

一句话核心

TL;DR

JSON Schema（JavaScript Object Notation Schema，JS 对象表示法的模式）是一份结构契约：它用机器能读的方式规定数据"必须有哪些字段、什么类型、什么取值范围"。它能被程序自动校验，不合规直接打回（像表单的必填校验），因此特别适合约束大语言模型的输出结构。但要记住：它只保证"结构对"，不保证"内容对"——胡编的合法数值照样能过校验，仍需业务规则兜底。

▸① Schema = 结构契约：字段 / 类型 / 取值范围机读契约
讲了什么

Schema（模式 / 规格）是一份结构契约：它规定一份数据或一段输出必须有哪些字段、每个字段是什么类型、取值落在什么范围。代表实现是 JSON Schema（JavaScript Object Notation Schema，JS 对象表示法的模式）。它本身也用 JSON 书写，因此几乎任何编程语言都能读懂，是一种跨语言通用的校验方案。

通俗解释

把它想成一张"表格的填写规范"：姓名必填且是文字、年龄必填且是 0–150 的整数、邮箱要符合邮件格式。规范本身不是数据，而是描述"合格数据长什么样"。

与你系统的关系

在 S31 里，每类资产都配一份 schema：schema_raw 管原料、schema_wiki 管词条、schema_output 管交付物……让每条 K-IR（Knowledge Intermediate Representation，知识中间表示）单元都有统一、可校验的字段结构。

▸② 能被程序自动校验，不合规直接打回自动门禁
讲了什么

Schema 的灵魂在于"给机器读、给系统校验"，而不是给人看的手册。程序可以拿 schema 去自动验证一份数据：合规就放行，不合规就直接拦下、打回，根本进不了下一步。整个过程不依赖人的自觉。

通俗解释

就像网页表单的"必填项校验"：你少填一栏、格式不对，提交键一按就被红字顶回来，根本提交不了。Schema 就是这套自动门禁，在数据入库前先过一道闸。

与你系统的关系

这正是 S31 里 Harness（马具 / 约束执行外壳）的关键一环：AI 生成的草稿必须先过 schema 校验再入库，不合格就回炉重来。校验 + 重试，是给"乱跑的马"套上的缰绳。

▸③ 用来约束大语言模型（LLM）的输出结构束住 AI
讲了什么

在 LLM（Large Language Model，大语言模型）场景里，Schema 用来强制模型吐出结构可预测的输出——比如必须是一个含特定字段的 JSON 对象，而不是一段自由发挥的散文。这样下游程序才能稳定地解析和使用模型的产出。讲话者还提出："把 AI 能做什么也写进 schema"。

通俗解释

不给 schema，模型像"想怎么答就怎么答"的话痨；给了 schema，等于发给它一张"必须照着填的表"。填出来的东西机器就能直接收下，不用再人肉去猜格式。

与你系统的关系

对应 S31 把 AI 限定成"通过 K-IR 与系统交互的加工助手"：它能写的字段、能改的状态，都被 schema_harness 白名单框死。Schema 既约束"输出长什么样"，也约束"AI 被允许做什么"。

▸④ 只保证"结构对"，不保证"内容对"能力边界 · 风险
讲了什么

Schema 有个硬边界：它只能验证结构合法性，不能验证内容真实性。一个字段类型完全合法、范围也没越界，但数值是 AI 胡编的，照样能顺利通过校验。结构校验之外，仍然需要业务规则与人工抽检来兜底。

风险边界

别把"过了 schema 校验"误当成"这条数据是对的"。校验通过只意味着"格式没毛病"，幻觉内容可以穿着合法格式的外衣蒙混过关。务必再加一层语义/业务层的检查，关键数据还要人审。

与你系统的关系

这解释了为什么 S31 里 Schema 不能单打独斗，要和 Harness 的重试、最小权限护栏、E 区审计并存。结构闸门挡住"乱来的格式"，业务规则与人审挡住"合法外衣下的假内容"。

▸⑤ Schema-as-Code：规范从"文档"变"可执行规则"范式升级
讲了什么

"Schema-as-Code"承袭 Everything-as-Code / Policy-as-Code（一切皆代码 / 策略即代码）思潮：规则不再写成给人看的 Word 文档，而是写成能被程序执行和校验的代码或配置。一句"把 SCHEMA.md 变成可执行规则"，意思就是——规范不再是摆设，而会真的把不合规的东西拦下来。

通俗解释

区别在于"贴在墙上的规章"和"刷卡才开的闸机"。前者靠员工自觉，没人看也就形同虚设；后者不合规就是过不去，规则被真正强制执行了。

与你系统的关系

S31 把每条规范写成 schema 字段或校验脚本，接到提交 / 入库环节自动跑、违规即报错。风险提醒：可执行规则也要维护与版本管理，规则写死又没人更新，就会变成挡路的"僵尸闸机"。

把 JSON Schema 接回你的 S31 系统

在主页里，"Harness 和 Schema 负责把 AI 幻觉束住"是一句口头直觉；"空度"提议把它展开成五张各管一段的 schema。一张对照说清分工：

五张 Schema 薄纸：各管一段，互不打架

schema_storage 管目录边界与命名；schema_raw 管来源与原料状态；schema_wiki 管词条字段与知识编辑；schema_output 管交付物的类型 / 版本 / 受众；schema_harness 管状态转换、通知与流程规则。这是"单一职责 + 关注点分离"的体现——一份大而全的规则，被拆成五份各管一摊的小契约。

AI 生成草稿 → JSON Schema 校验（结构闸门）→ 业务规则 / 人审（内容闸门）→ 合格入库 ／ 不合格回炉
▸Schema 与 Harness、FSM 怎么配合组合拳
Schema 的角色

定义"合格长什么样"——字段、类型、取值，以及"AI 被允许做什么"的白名单。它是静态的契约。

Harness 的角色

把 Schema 用起来：每步只允许特定动作，输出过校验，不合格就重试。它是动态的执行外壳。

和 FSM 的关系

schema_harness 还定义"哪个状态能转到哪个状态"，这正是 FSM（Finite State Machine，有限状态机）的转移表。Schema 立规矩、Harness 执行、FSM 管流转。详见 有限状态机专题 → 与 主页「空度」提议 →
