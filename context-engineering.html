in one line

一句话核心

TL;DR

提示词工程是"怎么把一句话问得更好"；上下文工程（Context Engineering）是"在模型生成答案前，系统地决定它该看到哪些信息"。因为上下文窗口是稀缺资源，塞太多会"降智"、太少又没依据，所以要靠检索、记忆、压缩、工具编排等手段动态管理"模型每一步看到什么"。落到实处，就是用 context.md 这类持久记忆解决大语言模型（Large Language Model，LLM）"开新窗口就失忆"的痛点，并对内容做分层与摘要，只喂当前任务真正用得上的部分。

▸① 从提示词工程，升级到上下文工程范式升级
讲了什么

提示词工程（Prompt Engineering）关注"怎么把一句指令写得更好"；上下文工程（Context Engineering）把视野放大到——在模型生成前，用一套系统去组织好它能看到的全部信息：指令、相关资料、历史记忆、工具返回的结果等。业界普遍把"动态、自动地为模型组织上下文"视为构建 AI 应用的头号工作。

通俗解释

提示词工程像"把问题问清楚"，上下文工程像"开考前帮考生把对的参考书、笔记、工具都摆到桌上"。问得好固然重要，但桌上摆什么，往往更决定答得好不好。

与S31的关系

你们早就在做这件事却没命名：context.md、执行手册、网页步骤书，本质都是"给 AI 摆好这一步该看的东西"。把它们统一理解成"上下文工程"，就能更系统地设计，而不是零散地堆文档。

▸② 上下文窗口是稀缺资源：管理"模型每步看到什么"核心约束
讲了什么

模型的上下文窗口（一次能读入的 token 数）是有限且有成本的。上下文工程的第一性原理是：窗口里每一格都很贵，要精挑细选放什么。塞进太多无关内容，不仅烧钱，还会稀释模型注意力、把关键信息淹没，反而表现下降。

通俗解释

像桌面就那么大：摊一两本翻开的书还能高效查；把整个书架全堆上去，反而找不到要用的那一页。信息不是越多越好，是越"对"越好。

与S31的关系

这就是为什么 context.md 不能无限长。要像管预算一样管上下文：每轮只装"对当前任务有用"的那部分，把其余归档——和你们 K-IR"只留稳定可复用单元"的取舍是同一逻辑。

风险/边界

别迷信"超长上下文窗口"能一劳永逸。窗口越长，无关信息干扰与成本越高；"能放下"不等于"该放下"。

▸③ 四大手段：检索、记忆、压缩、工具编排方法工具
讲了什么

上下文工程靠几类手段动态组织信息：检索——用 RAG（Retrieval-Augmented Generation，检索增强生成）临时取回相关资料；记忆——保存并调取长期/短期信息；压缩——对长历史做摘要、把臃肿内容收紧；工具编排——通过 MCP（Model Context Protocol，模型上下文协议）等方式按需调用工具，并把返回结果恰当地装回上下文。

通俗解释

像给 AI 配了个能干的助理：需要资料就去查（检索）、重要的事记下来（记忆）、长文先划重点（压缩）、该用工具就去用并把结果端回来（工具编排）。

与S31的关系

对号入座：云端记忆≈记忆 + 检索；执行手册/SOP≈把"该调哪个工具、按什么步骤"固化下来的工具编排；context.md 的摘要归档≈压缩。你们的落地方法，正是这四大手段的手工版。

▸④ context.md：持久记忆，治"开新窗口就失忆"持久记忆
讲了什么

LLM 默认没有跨会话记忆：开一个新窗口，它就忘了上次聊过什么。context.md 是一份手工维护的"长期记忆 / 项目交接文档"：每轮把"问题 + 关键结论 + 下一步"追加进去，新会话开头先把它喂回模型，让对话能"接上一次的进度"。

通俗解释

像给接班同事留的交接本：他不用从头问，翻一遍就知道项目来龙去脉、为什么这么决定、现在到哪一步了。

与S31的关系

这正是 theme 06 的核心工具，也是 K-IR 思路在"人机协作层"的轻量版：先用一个 md 文件把记忆沉淀起来跑通，再考虑做成结构化的后台中间层。详见 主页 theme 06 →

▸⑤ 别把上下文塞爆：要分层与摘要风险与治理
讲了什么

context.md 这类记忆会越长越臃肿，最终塞爆上下文窗口、反而"降智"。对策是分层管理：保留高层摘要供日常使用，把细节归档到另一处需要时再取；定期压缩、剔除过时内容，只让当前任务相关的部分进入窗口。

通俗解释

像整理笔记本：常翻的写在首页提要，啰嗦的细节挪到附录。不是不留，而是分层收纳，用时再翻附录。

与S31的关系

这一条直接定义了 context.md / 云端记忆的"维护 SOP"：定期摘要 + 归档。同时注意云端记忆的隐私、数据归属与可迁移性——重要资产要回流到自己掌控的本地/仓库。

风险/边界

账号、手机号等敏感信息别写进任何会被提交或上传的上下文文件；平台一旦关停或改政策，云端记忆可能带不走。

connect

把上下文工程接回你的 S31 系统

这页补上的是 theme 06 那套落地工具背后的统一名字与原理。一句话对照：

上下文工程（人机协作层）　↔　K-IR（后台知识工程层）

两者都在解决同一个问题——在有限的"工作内存"里，只放最该看的信息。context.md / 执行手册 / 云端记忆是这套思路的手工、轻量版，靠人维护一份份文档；K-IR 则是把它结构化、自动化的后台中间层。先把轻量版跑顺，再决定哪些值得升级成 K-IR。

▸落地清单：在 S31 里怎么做上下文工程怎么做
建记忆

每轮对话结束，让 AI 把"问题 + 关键结论 + 下一步"追加进 context.md；新会话开头先喂回去。

控容量

定期对 context.md 做摘要 + 归档，只保留对当前任务有用的部分，防止塞爆窗口降智。

配工具

把"该查什么、该调哪个工具、按什么步骤"写进执行手册/SOP（工具编排）；高频汇总放云端，定稿回流本地长期保存。

守边界

敏感信息不入库；重要记忆务必有一份在自己掌控的仓库里。详见 主页 theme 06 →
