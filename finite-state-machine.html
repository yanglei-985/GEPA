TL;DR

▸① 任一时刻只在一个状态，靠"事件"做合法转移基本定义
讲了什么

FSM（Finite State Machine，有限状态机）是一种抽象计算模型：系统在任一时刻只处于有限个明确状态中的某一个。当某个"事件"（外部输入）发生，系统才按预先定义的合法路径从当前状态转移到下一状态——没定义的转移一律不允许。它由状态、事件、转移三要素构成。

通俗解释

像一台只认特定按钮的机器：它现在停在某个挡位，你按下某个键它才换到另一个挡位，乱按无效。"有限"是指挡位数量是固定可数的，不会无限多。

与你系统的关系

这正是 S31 把"固定流水线"升级成状态机的理由：每个知识单元用一个 status 字段表示当前状态，只能按转移表往前走。它把流程变成"可分叉、可回退、可审计"的受控过程。

▸② 两种表达：状态图 与 转移表怎么画
讲了什么

一台 FSM 通常有两种等价的表达方式：状态图（state diagram）——用圆圈表示状态、带箭头的线表示转移，直观好看；转移表（transition table）——用表格列出"当前状态 × 输入事件 → 下一状态"，便于程序实现和穷举检查。视频里也提到 Mealy 机与 Moore 机两种经典模型，区别在于输出是看"状态+输入"还是只看"状态"。

通俗解释

状态图像"地铁线路图"，一眼看清能从哪站到哪站；转移表像"时刻表"，把每种情况列得清清楚楚。一个给人看，一个给机器查。

与你系统的关系

在 S31 里，schema_harness 写的就是一张转移表：定义"哪个状态能转到哪个状态、由谁触发、转移时通知谁"。画一张状态图贴在文档里，团队沟通流程就有了共同语言。

▸③ 物流类比：已揽收→运输中→派送中→已签收，不能跳站直觉类比
讲了什么

快递物流就是一台天然的 FSM：包裹状态从已揽收 → 运输中 → 派送中 → 已签收逐站推进。你随时能查到"包裹现在在哪一站"，而且不可能从"已揽收"直接跳到"已签收"——必须按合法路径一步步走。

通俗解释

这就是 FSM 的精髓："可观测"（随时知道在哪）+ "受约束"（不能乱跳）。任何有"阶段"的事情——订单、审批、游戏角色、红绿灯——都能套这个模型。

与你系统的关系

把知识单元当成"包裹"：它从原料一路被加工成交付物，每一步都有据可查、且不能越级跳。这正是 S31 想要的"每个单元都知道自己在哪儿、下一步是什么、是否有交付"。

▸④ 知识单元生命周期：从 raw 到 archived 的七态S31 状态链
raw_unprocessed → ir_draft → ir_reviewed → wiki_generated → output_draft → output_reviewed / delivered → archived
讲了什么

S31 把每条知识单元的一生建成一台 FSM：raw_unprocessed（原料未处理）→ ir_draft（中间表示草稿）→ ir_reviewed（中间表示已审）→ wiki_generated（词条已生成）→ output_draft（交付物草稿）→ output_reviewed / delivered（交付物已审 / 已交付）→ archived（归档）。每一步都是一个合法转移，由人、AI 或脚本按规则触发。

通俗解释

这里的 IR 就是 K-IR（Knowledge Intermediate Representation，知识中间表示）。整条链就是"毛坯料 → 半成品 → 成品 → 入库"的工厂流水线，每个工位都有明确的"上一道/下一道"。

与你系统的关系

这条状态链同时贯穿三个专题：PARA 在文件夹层描述"项目→归档"的流动，FSM 把它落成可审计的转移，Schema 给每个状态定义合格字段。三者合起来就是后台中枢。参见 PARA 专题 → 与 JSON Schema 专题 →

▸⑤ 别让状态/转移爆炸：先少后多设计风险
讲了什么

FSM 设计有个常见陷阱：状态和转移一旦贪多，就会爆炸。几十个状态彼此都能跳，转移路径呈指数增长，维护和调试会变成噩梦。正确做法是先用少量主状态把主干跑通，再按真实需要逐步加分支。

风险边界

状态爆炸常源于"把每个细微差别都设成新状态"。很多差异其实可以用"状态 + 附加字段"表达，而不必新增状态。复杂场景可考虑分层状态机（hierarchical FSM）把相关状态分组收纳。

与你系统的关系

呼应 S31 反复强调的"先能跑通，再谈扩展"：上面那 7 个主状态足够先跑起来，别一上来就设计几十个状态。等真实素材流过、暴露出确实需要的分支，再谨慎扩展。

connect

把 FSM 接回你的 S31 系统

在主页里，"把固定流水线改成状态机"是一句口头直觉；FSM 正是把它工程化的引擎。一句话对照：

FSM Harness：让每个知识单元都"有自我意识"

把 FSM 装进 Harness（马具 / 约束执行外壳），每个知识单元就能回答三个问题：我在哪儿（当前 status）、下一步去哪（转移表里的合法路径）、是否有交付（是否到达 delivered）。FSM 立流程、Schema 立字段、Harness 负责执行与重试——三者合起来，就把"乱跑的 AI"约束成"按流水线推进的加工助手"。

▸FSM 与 DFA：顺带厘清一个常被混的术语术语澄清
FSM 与自动机

在计算理论里，有限状态机也叫有限自动机（finite automaton）。它分两类：DFA（Deterministic Finite Automaton，确定性有限自动机）——同一状态遇同一输入只有唯一去向；以及非确定性的版本（同一输入可能有多个去向）。

对工程的意义

S31 的流程状态机应当是确定性的：同一状态、同一事件，结果必须唯一可预测，否则流程会变得无法调试。换句话说，业务用的状态机基本都按 DFA 来设计。

怎么用

记住一句：流程要可预测，就用确定性转移。把"当前态 + 事件 → 唯一下一态"写死进 schema_harness，别留模棱两可的分叉。详见 主页「空度」提议 →
