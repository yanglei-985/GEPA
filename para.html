in one line

一句话核心

TL;DR

PARA（Projects 项目 / Areas 领域 / Resources 资源 / Archives 归档）的精髓只有一句：按"可行动程度"分类，而不是按"学科主题"分类。任何笔记、文件、链接，先问"它现在能不能推动某件正在做的事"，再决定放进哪一类；项目一完成就移进归档。它是一套灵活的方法，不是必须死守的律法——别为"这到底算领域还是资源"而内耗。

▸① 四大类：Projects / Areas / Resources / Archives四类定义
讲了什么

PARA（Projects / Areas / Resources / Archives，项目 / 领域 / 资源 / 归档）把一切数字信息收进四个顶层文件夹：Projects 项目＝有截止线、有明确成果的活跃任务；Areas 领域＝需要长期维护、没有终点的责任范围；Resources 资源＝当下用不上但将来可能参考的主题资料；Archives 归档＝已完成或不再活跃、暂时封存的内容。

通俗解释

像厨房收纳：正在做的菜放灶台（项目）、常用调料放手边（领域）、菜谱书上架（资源）、过季工具进储藏室（归档）。四个抽屉，每样东西都有家可回。

与你系统的关系

对应 S31 仓库的顶层目录设计：先建这四个文件夹当"物理骨架"，再让 raw / wiki / output 等内容各归其位。骨架先立稳，后面的 K-IR（Knowledge Intermediate Representation，知识中间表示）、状态机才有地方安放。

▸② 精髓：按"可行动程度"分类，而非按学科主题分类原则
讲了什么

PARA 最反直觉、也最关键的一点：不按"它是什么主题"归类，而按"它离行动有多近"归类。同一个主题（比如"写作"）的材料，可能同时散落在项目、领域、资源里——因为有的正在用、有的长期维护、有的只是备查。判断依据始终是"这条信息现在能不能推动某件具体的事"。

通俗解释

传统分类像图书馆按学科上架（历史区、科学区），找资料得先想"它属于哪门学问"。PARA 反过来按"现在用不用"摆：手头要用的放最近，越不用的越往里收。是按"用途"而非"身份"整理。

与你系统的关系

这正呼应 S31 里"价值要留下脚印"的思路：知识不是按学科冻结存档，而是按它当前的可用性流动。它和后台的 value_level（价值级别）字段是同一种"以行动/价值为中心"的世界观。

▸③ 动态流动：项目一完成就移到归档生命周期
讲了什么

PARA 不是装好就不动的柜子，而是会持续流动的系统。一个项目做完，就立刻把它从 Projects 移到 Archives，给活跃区"腾地方"；当归档里的旧材料又被新项目需要时，再把它"激活"调回来。东西的归属会随它的活跃度变化而变化。

通俗解释

像办公桌：做完的项目资料别堆在桌面占地方，收进抽屉（归档）；桌面只留正在推进的事。需要时再从抽屉翻出来。保持"活跃区清爽"是关键。

与你系统的关系

这条"流动"直接对应 S31 的 FSM（Finite State Machine，有限状态机）生命周期——知识单元从 draft 一路走到 archived。PARA 在文件夹层面描述这种流动，状态机则把它工程化为可追踪、可审计的转移。详见 有限状态机专题 →

▸④ 它是"方法"不是"律法"：别陷入分类内耗使用心法 · 风险
讲了什么

PARA 是一套灵活的组织方法，而不是必须逐字死守的"范式律法"。它的目标是减少摩擦、让你更快行动，而不是给你新增一道"必须分对类"的考试。Forte 本人也强调它不是固定范式，可按个人习惯裁剪。

风险边界

最大的坑是过度纠结"这到底算 Area 还是 Resource"——为分类完美而反复挪动、犹豫，结果分类内耗反而拖垮了产出。记住：分得"够用"就好，把时间留给真正的工作，模糊时先扔进最接近的一类，以后再说。

与你系统的关系

对应 S31 反复强调的"先能跑通，再谈优雅"。设计目录骨架时别一上来追求完美分类法；先用 PARA 粗粒度跑起来，等真实素材流过、痛点暴露了，再细化规则。

▸⑤ 怎么落地：仓库顶层就建四个文件夹实操
讲了什么

落地极简：在笔记库 / 代码仓库的顶层直接建 Projects、Areas、Resources、Archives 四个文件夹。每来一条新东西，先问一句"它现在可行动吗"，再决定归位。项目结束就从 Projects 移到 Archives。

通俗解释

不需要复杂工具，记事本、文件夹、Obsidian 都行。关键是养成那个"归位前先问一句"的小习惯，而不是先设计一套庞大的目录树。

与你系统的关系

在 S31 里，PARA 这四个文件夹就是承载 raw / wiki / output / harness 等内容的物理容器；而每个文件里的 YAML（YAML Ain't Markup Language，一种人类易读的结构化数据格式）frontmatter 则记录状态字段——文件夹管"放哪"，YAML 管"是什么状态"。

connect

把 PARA 接回你的 S31 系统

在主页「组合拳」那章里，五个关键词各管一段。PARA 管的是最底层的"物理摆放"——其它机制都建在它之上。一张对照表说清分工：

PARA（物理骨架）　↔　YAML（骨架内的树）　↔　K-IR（缓冲层）

PARA 决定"东西放在哪个文件夹"（粗粒度的物理位置）；YAML（YAML Ain't Markup Language） 在每个文件头部记录字段与状态，是"骨架内部的树"，让分类信息被机器看见；K-IR（Knowledge Intermediate Representation，知识中间表示） 则是 Raw 与 Wiki 之间的缓冲单元。三者分层配合：PARA 摆位置、YAML 标状态、K-IR 做加工缓冲，互不抢活。

▸为什么 PARA 是"骨架"而不是"血肉"职责定位
骨架的角色

PARA 只负责大块的物理分区，回答"这东西大致归哪个区"。它不关心内容细节，就像房子的承重墙不关心你怎么摆家具。

血肉由谁管

具体内容的结构化、字段校验、流转，交给 K-IR、Schema、FSM 状态机。在主页比喻里，"骨头排列组合"是 PARA 这类框架，"血肉填进去"才是内容加工。

怎么用

先用 PARA 把仓库顶层骨架立起来，再在各文件夹内部用 Schema 与状态机管理生命周期。详见 主页「组合拳」章 → 与 JSON Schema 专题 →
