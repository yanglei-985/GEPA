▸① 在终端里运行的 AI 编码代理：读代码、改文件、跑命令、用 Git是什么
讲了什么

Claude Code 是一个住在终端（命令行）里的 AI 编码代理。它不只是聊天，而是真的能在你的项目里动手：读懂代码库、编辑文件、执行命令与测试、并通过 Git 完成提交等版本操作。它运行在 CLI（Command-Line Interface，命令行界面）里，也有 IDE 集成与桌面端等形态。

通俗解释

像请了一位能直接上手敲键盘的工程师助手坐在你的电脑前：你说需求，它去翻文件、改代码、运行验证，而不是只给你一段建议让你自己粘。

与你系统的关系

正是主页主题 07 那句"交给 AI 解构项目"的代表工具：拉下仓库后，让它通读并产出"它是干嘛的、怎么跑、关键文件在哪"的解构笔记。

▸② 不是自动补全，而是项目级"自主执行"——给目标它干活代理 vs 补全
讲了什么

传统的代码补全是"你打一半，它接下半句"；Claude Code 是代理（agent）：你给一个目标（"修这个 bug""加这个功能"），它会自己规划步骤、跨多个文件改动、运行验证、必要时反复迭代，直到达成目标。区别在于它处理的是"任务"而非"下一个字符"。

通俗解释

补全像"输入法的联想词"，代理像"派活给一个会自己干完的实习生"——你说要什么结果，过程它自己安排。

与你系统的关系

这种"给目标、自主执行"恰是 S31 后台想要的能力，但也正因为自主，才更需要 Harness（马具/护栏）把它束住——见第⑤条。

风险边界

自主≠万能：任务越大越模糊，它越可能跑偏或做多。把目标拆小、给清验收标准，比一句"帮我把项目重构好"靠谱得多。

▸③ 靠 CLAUDE.md / 上下文文档接上项目历史上下文
讲了什么

代理要干好活，得"知道这个项目的来龙去脉"。Claude Code 约定用一个 CLAUDE.md 文件（放在项目里的上下文/规则文档），记录项目结构、约定、注意事项；它也会读代码库本身来建立上下文。还能通过 MCP（Model Context Protocol，模型上下文协议）这类开放协议，接上外部数据源与工具。

通俗解释

像给新来的同事一份"入职手册"：项目怎么跑、规矩是什么、哪些雷不能踩。手册越清楚，他上手越快、越不容易闯祸。

与你系统的关系

直接呼应 S31 反复强调的"上下文沉淀 / 说明手册"：CLAUDE.md 就是把 BNY 教的"context.md（项目上下文文件）"思路落到 Claude Code 上的具体形态。

风险边界

CLAUDE.md 与上下文文档会被读取，别把账号、手机号、密钥等敏感信息写进去，更别让它进入会被提交的仓库。

▸④ 适合"拉项目 → 交 AI 解构 → 落地"的工作流工作流
发现仓库 → git clone 拉到本地 → Claude Code 读懂并解构 → 改造/加功能 → 跑通并接进流程
讲了什么

Claude Code 正好补上主页主题 07 能力链的"解构与改造"那一段：项目用 Git 拉到本地后，交给它通读、产出解构笔记、再按目标改造，直到跑通。它把"开源为我所用"从"看得懂"推进到"动得了手"。

通俗解释

GitHub 把"原料"搬进门，Claude Code 是"会加工原料的师傅"——把别人的项目拆开、看明白、改成你要的样子。

与你系统的关系

对接主页 「下一步」：①把 Claude Code 用起来、②把"GitHub 搜索→拉项目→交 AI 解构"练熟。两者本就是配套的一条链。

▸⑤ 管好权限与安全：别让它乱删乱改，敏感信息别入库权限护栏
讲了什么

能改文件、能跑命令，意味着也能误删、误改、跑出有副作用的命令。所以要给它护栏：按最小权限原则授权、关键操作要人确认、重要改动走 Git 分支以便回退、绝不把密钥/口令提交进仓库。自主能力越强，护栏越要先立。

通俗解释

像给实习生权限：能写代码、能跑测试，但"删库、动正式数据、改制度"的钥匙不能随手给——先划好红线，再放手。

与你系统的关系

这正是 S31 的 Harness（马具）与 AI Guard（权限护栏）思想在工具上的落地：用状态机 + 校验 + 最小权限把"乱跑的马"束住。详见 主页主题 05。

风险边界

别图省事开"全自动免确认"：在不熟悉的仓库或有生产影响的环境里尤其危险。先在可回退的分支 / 沙箱里试，确认无误再放权。
